# 計画レビュー（外部エージェントによる批判的レビュー）

このリファクタリング計画（[`00-analysis.md`](./00-analysis.md) / [`01-roadmap.md`](./01-roadmap.md) /
[`prompts/`](./prompts/)）を、実コードと照合した上で**是非も含めて批判的に**レビューした結果を記録する。

## このレビューを受けて決まったこと（要約）
1. **全面移行は ROI が低い**との指摘を受け、**テスト安全網タスク [`prompts/TS-test-safety-net.md`](./prompts/TS-test-safety-net.md) を新設**し、
   リスクの高い移行（T01/T03/T05/T09）の**前提**とした。
2. **挙動不変を検証する手段が無い**点が最大の穴。TS で「テスト可能な層（シリアライズ・use_case・migration）」を固める。
   UI ロジック（widget 内）は抽出前はテスト不能なため、その移行は **move only・手動確認**に徹する。
3. **T09（migration）は最高リスク**。TS でゴールデンテストを用意してから着手する（または「触らない」も選択肢）。
4. 英語コメント統一・lint を最後に回す判断・並列時の生成物コンフリクト等の指摘は、別途 roadmap に反映検討。

---

## 1. 総合評価
**結論: 計画の「設計品質」は高いが、「規模に対する妥当性」に疑問あり。全面実行は推奨しない。投資対効果の観点で T07・T11・部分的な P2 修正・T13 に絞った "選択的実行" を強く推奨する。**

根拠: 分析（P1〜P7）は実コードとほぼ完全に一致しており、事実誤認や誇張は見当たらない（裏取り済み）。タスク分割・ストラングラー方針・Isar/Firestore 互換への注意も的確。しかし対象は**手書き14,342行・126ファイル・開発者1名**の個人アプリで、**実テストが事実上ゼロ**（`test/widget_test.dart` はデフォルトのカウンタ smoke test で、存在しないカウンタを参照しており実質壊れている）。この状態で13タスクに及ぶ大規模な構造移動を「挙動不変」で完遂する保証手段がない。最もリスクが高い T01〜T06・T09 ほど、それを守る安全網（テスト）が欠落しているという逆相関がある。

## 2. 是非についての評価
**価値があるのは一部のみ。全面移行は過剰設計の懸念が大きい。**

- **DDD/feature-first 全面移行のROIは低い**: feature 間境界の明確化・並列開発・オンボーディングといった DDD の主便益は、**複数人チーム**でこそ効く。作者1名・14k行の本アプリでは、移行コスト（13セッション、全 import 貼り替え、コード生成、回帰確認）に見合うリターンが乏しい。プロンプト自身が「リスクが高い箇所は無理に動かさず最小の抽出に留めてよい」（T02）と繰り返しており、**著者自身が便益の薄さを暗に認識している**。
- **一方で「明確に直す価値がある実害」は確かに存在する**:
  - **P2 のレイヤー違反（実在を確認）**: `lib/providers/digestive_providers.dart:10` がデータ層から `pages/home_tabs/tab_digestive_list.dart`（UI層）を import し、ドメインモデル `DigestiveWithSubmission` が page 内（`tab_digestive_list.dart:129`）に定義されている。循環依存の温床で、feature 化と無関係に**単独で修正すべき**負債。
  - **P6 デッドコード（memorize_card）**: `camera_preview_page.dart`(835行) は 100% コメントアウト、`tab_memorize_card.dart` も実行コード 0 行。削除は純粋にプラス。
  - **P7 EventBus**: 実利用は `SwitchBottomNav` 1種のみ（`main.dart:150` fire / `home_page.dart:133` listen）。残りは memorize_card 絡みのコメント。T07 後はほぼ自明な置換。
- **推奨する代替案（縮小版ロードマップ）**:
  1. **T07（memorize_card 削除）** — 即実施。リスク最小、スコープ縮小効果大。
  2. **P2 のピンポイント修正** — `DigestiveWithSubmission` を models 相当へ移し、data→UI 依存を切る。feature 全体移行は不要、この1点だけでも価値がある。
  3. **T11（EventBus 撤去）** — 実イベント1種なので低コスト。
  4. **T13（lint 一掃）** — 構造移行と独立に実施可能で、品質改善が明確。
  5. submission/digestive/timetable の**全面 feature 移行（T01〜T06, T09, T10）は「やらない or 機会があれば」に格下げ**。少なくとも安全網を入れるまで着手しない。

## 3. 良い点
- 分析の**事実精度が高い**。P1（submission の分裂）、P2、P5（`submission_editor.dart`=実測385行）、P6 すべて実コードと一致。
- **ストラングラー方式・「挙動を変えない」原則・feature 単位の完結**という移行思想は教科書的に正しい。
- **Isar スキーマ／Firestore キー名の不変条件**を全プロンプトで繰り返し明記している点は、本アプリ最大の破壊リスクを正しく押さえている。
- **T13 の lint ベースライン実測**（338 issues、async系95%、ワースト10ファイルまで）が具体的。
- T00 が既に**実際にコミット済み**で、計画が机上で終わっていない。
- プロンプトが**自己完結**しており、別セッションへの引き継ぎ設計（`_TEMPLATE.md`）が現実的。

## 4. 問題点・懸念（重大度付き）
- **[重大] 挙動不変を担保する安全網が皆無**（全タスク／特に T01〜T06, T09）。実テストはゼロ。DoD「既存テストが通る（存在する範囲で）」の範囲が空集合。全 import 貼り替え＋ロジック抽出を回帰検出なしで「挙動不変」と称するのは最大の穴。コンパイルが通る≠挙動不変。
- **[重大] T13 と各移行タスクの依存が二度手間を生む**。「移設で違反が再発するから lint は最後」の裏返しで、移設のたびに338件を引きずる。`discarded_futures`/`unawaited_futures`（260+62件）は**挙動に直結**。散らばった後にテスト無しで「1件ずつ挙動確認」は検証不能。
- **[高] T09（data_sync migration）のリスク評価が楽観的**。`data_sync_service.dart` は全件同期＋`_migrate`（v4→7）を279行に混載。mapper を feature 側へ動かした後に触るため、所在変更と migration の文字列処理が同時に絡む。**「触らない」が最も安全**。
- **[中] 「1セッションで完結」が甘いタスク**（T02, T13）。T02 は385行のロジック抽出＋移設＋ルーティング＋生成。T13 は330+件の手動 async 確認。途中で analyze が中途半端だと DoD を満たせず次セッションが汚れを引き継ぐ。
- **[中] CI / 検証環境への言及が薄い**。各セッションが analyze/build_runner を実走できる保証がないと DoD「analyze クリーン」が確認不能。
- **[中] T12 の依存境界 lint が「custom_lint かスクリプトか」未決**。再発防止の核が選択肢提示止まり。grep スクリプトだと IDE で効かず実効性が落ちる。
- **[低] 英語コメント統一方針の便益が不明**。作者1名（日本語話者）の個人リポでは差分肥大のコスト増。挙動不変リファクタ中に言語を変えるのは signal とノイズを混ぜる。

## 5. 具体的な改善提案
1. **順序を組み替える**: 価値が高く独立・低リスクなものを先頭へ。推奨: T07 → P2ピンポイント修正 → T11 →（T13 機械修正分の先行）→ 必要ならドメイン移行。
2. **着手前に最小限の安全網**: (a) `data_sync_service` migration のゴールデンテスト、(b) submission 保存フローの use_case テスト。これ無しの T01/T02/T09 は「挙動不変」を名乗れない。
3. **T13 を分散 or 独立先行**。async lint は挙動直結のため、移設で散らす前に決着させる方が安全。
4. **T09 は「migration 凍結（触らない）」をデフォルトに**。feature 化対象から外す選択肢を明記。
5. **英語コメント統一を本移行から外す**。やるなら挙動変更を伴わない単独 PR で。
6. **CI を前提条件として明示**。各セッションが analyze/build_runner を実走できる環境を先に固定。
7. **「やらない」基準を文書化**。「最小の抽出に留めてよい」を明示的な撤退ライン（例: テスト不在領域のロジック抽出は禁止、移動のみ可）へ格上げ。

## 6. 見落とされているリスク
- **回帰検出手段の不在そのもの**（最大の盲点）。計画全体が「analyze クリーン＝OK」依存だが、analyze は型と lint しか見ない。fire-and-forget の発火順・banner 表示条件・pop タイミングを機械検証する手段が皆無。
- **`DigestiveWithSubmission` の波及**: 16箇所で参照。models へ移すと submission feature の公開クエリへの依存が新たに生じ、**features 間参照禁止ルールの例外**を早くも作る。線引きが曖昧だとルールが空文化。
- **コード生成のコンフリクト**: 並列セッション推奨だが、`*.g.dart`/`*.freezed.dart` の全体再生成で広範囲に差分が出る。digestive と timetable の並列は**生成物コンフリクトを高確率で起こす**。
- **`schemaVersion` 取り違え**: Isar schema list と Firestore schemaVersion が同一ファイルに同居。Isar を触る T01/T03/T05 側に「7 は Firestore 用」の再掲が無く、誤ると全ユーザーで不要な migration が走る破壊事故になりうる。
- **`link_events_provider` / `functions_service` の扱いが計画に無い**。どのタスクでどの feature へ寄せるか言及がなく、T12 の空ディレクトリ削除時に宙に浮く可能性。
